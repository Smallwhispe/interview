# Binary（二进制文件）

## 1. 执行标准

-   **核心原则**：为保证代码可读性与可审计性，TTP（Texas Trusted Platform）机房下的所有构建行为都必须发起自源码。因此，**严禁在 Git 仓库中直接提交任何可执行的二进制文件、预编译库或构建产物**。
-   **适用范围**：此规则覆盖所有参与 TikTok App 编译打包的组件、工具和代码库，包括但不限于 `.jar`, `.aar`, `.so`, `.a`, `.pyc`, `.o`, `.class`, `.exe` 等文件。

### 1.1 扫描范围（文件后缀清单）

#### 1.1.1 待扫描的二进制文件后缀

以下后缀的文件均纳入二进制扫描范围（大小写不敏感）：

| 类别 | 后缀清单 |
| :--- | :--- |
| **预编译库 / 链接库** | `*.a`、`*.so`、`*.so_pre`、`*.o`、`*.lib`、`*.dylib`、`*.framework`、`*.syso` |
| **JVM 产物** | `*.jar`、`*.aar`、`*.class`、`*.dex`、`*.war` |
| **可执行 / 安装包** | `*.apk`、`*.app`、`*.bin`、`*.dll`、`*.ipa`、`*.iso`、`*.exe`、`*.bundle`、`*.dos` |
| **Python 中间产物** | `*.pyc` |
| **镜像 / 原始数据** | `*.raw` |
| **压缩包** | `*.7z`、`*.gz`、`*.gzip`、`*.tar`、`*.z`、`*.zip`、`*.zx`、`*.tgz`、`*.xz`、`*.bz2` |

> 备注：上述清单参考神盾"二进制文件管控"扫描规则，与 `scripts/` 中的扫描脚本保持一致；新增类型时需同步更新本清单与脚本。

#### 1.1.2 无后缀名文件的扫描策略

-   **强制纳入扫描**：对于**无后缀名**的文件，无法仅凭文件名判定类型，需进一步识别其是否为二进制：
    -   优先使用 `file <path>`（依赖 libmagic）或读取文件头 magic bytes 判别 MIME 类型；
    -   命中以下任一特征视为二进制并纳入告警：
        -   ELF（`7F 45 4C 46`）、Mach-O（`CA FE BA BE` / `CF FA ED FE` 等）、PE/COFF（`MZ`）等可执行文件头；
        -   归档/压缩魔数（如 `PK\x03\x04`、`1F 8B`、`7z\xBC\xAF\x27\x1C`）；
        -   非 UTF-8 可解码且不可打印字符占比超过阈值（建议 > 30%）。
-   **可读文本豁免**：若 `file` 输出为 `ASCII text`、`UTF-8 Unicode text` 等纯文本类型，或文件位于 `1.1.3` 的排除清单中，则不计入二进制问题。

#### 1.1.3 排除清单（不计入二进制问题）

以下 glob 模式命中的文件**不参与**二进制扫描，即使后缀匹配 `1.1.1` 也直接跳过：

| 类别 | 排除模式 | 说明 |
| :--- | :--- | :--- |
| **构建工具** | `*gradle-wrapper.jar` | Gradle Wrapper 必备工件，业界标准实践 |
| **VCS 元数据** | `*/.git*` | `.git/` 目录及 `.gitignore`、`.gitattributes` 等元数据文件 |
| **资源数据** | `*.json.xz`、`*.mmdb` | 业务运行时资源（如配置数据、IP 地理库） |
| **图片素材** | `*.png`、`*.webp`、`*.jpg`、`*.jpeg`、`*.gif` | UI / 资源图片 |
| **音视频素材** | `*.mp[34]` | `.mp3`、`.mp4` 媒体文件 |
| **字体素材** | `*.ttf`、`*.otf`、`*.woff` | 字体文件 |
| **证书 / CRL** | `*.crt`、`*.crl` | TLS 证书与吊销列表 |

> 排除清单应与 `scripts/` 内置的 glob 白名单保持一致；新增豁免必须附带审计说明（来源、用途、为什么不能源码化），不允许直接在仓库内提交未声明的二进制白名单条目。

## 2. 处理方法

### 2.1 服务端处理方案

1.  **删除**：
    -   对于编译中间产物（如 `.pyc`, `.o`, `.class`）或 `output` 目录下的构建结果，应直接从仓库中删除，并将匹配模式（如 `*.pyc`）添加到 `.gitignore` 文件中。
2.  **修改引用方式（依赖代理）**：
    -   对于开源的第三方库，应在构建或编译过程中通过公司内部的代理源实时下载，而非直接存放在代码库中。各语言生态对应的代理地址如下：
        -   **PyPi**: `bytedpypi.byted.org`
        -   **Maven**: `maven.byted.org`
        -   **Go Module**: 使用默认设置，无需额外配置 `goproxy`。
        -   **NPM**: `bnpm.byted.org`
3.  **源码同步与 SCM 预构建**：
    -   对于内部产生的预编译库，需将源代码同步至 TTP，并在 SCM (Software Configuration Management) 或 Luban 平台维护构建流水线，在 TTP 环境内进行预构建。使用方在自己的流水线中按需下载这些预构建产物。
    -   **C++ 内部依赖**：使用 BPT (BytePlus Package Tool) 完成 `.a` 或 `.so` 的预编译库构建。
    -   **Java/Node/Python 内部依赖**：使用 Luban 平台完成 `Maven/NPM/PyPi` 包的构建和发布。

### 2.2 客户端处理方案

1.  **源码整改**：从源码层面将二进制文件删除或替换为源码依赖。
2.  **组件平台（Bits）升级与同步**：
    -   **添加 TTP 标签**：在 Bits 组件平台为相关组件添加 `TTP` 或 `TTP common` 标签。
    -   **触发 TTP 构建**：带有 TTP 标签的组件在字节内部每次升级版本时，都会自动触发一个在 TTP 机房内的对应构建任务。
    -   **同步 Codebase**：在 Bits 平台的“仓库同步”模块中，注册需要同步到 TTP 的代码仓库和分支，确保构建所需的代码在 TTP 环境中可用。

## 3. 常见问题与典型案例处理

-   **`maven-wrapper.jar`**：此文件不应提交到代码库。应在 `.mvn/wrapper/maven-wrapper.properties` 中配置从 `https://maven.byted.org` 下载，并将 `maven-wrapper.jar` 添加到 `.gitignore`。
-   **`.pyc`, `.o`, `.class` 文件**：这些是编译中间产物，必须删除，并通过 `.gitignore` 屏蔽。
-   **预编译库 (`.a`, `.so`)**：
    -   如果是未修改的开源库，建议使用 BPT 等工具进行源码管理和构建。
    -   如果是自研或改动较大的库，必须删除二进制文件，改为源码引用。
-   **开源二进制文件**：
    -   若公网可下载，可通过在构建时使用 `wget` 等方式从已加白的域名下载。
    -   若来源无法追溯，则必须删除或进行源码化重写。
-   **商业合作提供的二进制**：需提供来源、用途等证明，通过 Oncall 通道进行人工评估，可能获得特殊豁免。
